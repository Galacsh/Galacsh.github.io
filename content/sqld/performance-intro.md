애플리케이션의 성능이 저하되는 경우, 개발자가 구축한 SQL구문에 대해서만 책망을 하는 경우가 많이 있다. 물론 그럴 수도 있지만 근본적으로 디자인이 잘못되어 SQL구문을 잘 못 작성하도록 구성될 수밖에 없는 경우도 빈번하게 발생되고 있음을 기억해야 한다.

---

# 성능 데이터 모델링의 정의

> 성능 데이터 모델링이란 데이터 모델링 때부터 **정규화**, **반정규화**, **테이블 합**, **테이블 분할**, **조인 구조**, **PK**, **FK** 등 여러 가지 성능과 관련된 사항이 데이터 모델링에 반영될 수 있도록 하는 것으로 정의할 수 있다.

## 성능이란?

- 일반적으로 성능이라고 하면 데이터 조회의 성능을 의미함
  - 데이터입력/수정/삭제는 일시적이고 빈번하지 않고 단건 처리가 많음
  - 데이터조회의 경우는 반복적이고 빈번하며 여러 건을 처리하는 경우가 많음
- 단, 업무에 따라서는 입력/수정/삭제의 성능이 중요한 경우도 있음

## 성능 데이터 모델링이 필요한 경우

대표적으로 다음과 같은 경우가 있음

- 데이터 모델 구조에 의해 성능이 저하되는 경우
- 데이터가 대용량이 되어 성능이 저하되는 경우
- 인덱스 특성을 충분히 고려하지 않고 인덱스를 생성한 경우

## 다양한 성능 데이터 모델링

- 정규화/반정규화를 통해 성능 향상
- 인덱스의 특징을 고려한 컬럼 순서 변형
- 테이블의 수직/수평 분할

위 경우 외에도 데이터 특성 등에 따라 여러 방법이 있을 수 있음

---

# 성능 데이터 모델링 수행시점

> **성능 데이터 모델링은 빠를수록 좋다.**

- **빠를수록** 비용이 들지 않는다.
  - ex. 분석/설계 단계에서 대충하고 SQL 튜닝, 하드웨어 증설 작업은 추가적인 비용을 소진하게 함
  - 데이터의 증가가 빠를수록 성능저하에 따른 개선비용이 기하급수적으로 증가함
- 즉, **분석/설계 단계에서** 데이터베이스 처리 성능을 향상시킬 수 있는 방법을 고려해야 함
  - ex. 어떤 트랜잭션이 성능이 저하되면 안되는 특징을 가지고 있는 경우
    - 프로젝트 초기에 운영환경에 대비한 테스트 환경을 구현
    - 그곳에 트랜잭션을 발생시켜 실제 성능을 테스트
    - 데이터 모델의 구조를 변경해보며 가장 적절한 구조 디자인

---

# 성능 데이터 모델링 고려사항

1. 정확한 정규화

> 정규화 자체로 관심사 별로 데이터를 분산시켜 성능 향상 효과를 가져옴

2. 데이터베이스 용량 산정

> 어떤 엔터티(테이블)에 데이터가 집중되는지 파악할 수 있음

3. 데이터베이스 트랜잭션 유형 파악

> CRUD 매트릭스 or 시퀀스 다이어그램을 통해 파악하는 것도 좋은 방법이며 유형을 파악하여 SQL 문장의 조인 관계 테이블에서 데이터 조회에 사용되는 컬럼들을 파악하여 설계할 수 있음

4. 트랜잭션 유형에 따른 반정규화

> 테이블, 속성, 관계에 대해 포괄적인 반정규화의 방법을 적용

5. 이력 모델의 조정, PK/FK조정, 슈퍼타입/서브타입 조정 등을 수행

> - 대량 데이터가 처리되는 이력모델에 대해 성능 고려가 필요
> - PK/FK의 순서가 인덱스 특성에 따라 성능에 영향을 미치는 영향도가 크기 때문에 반드시 PK/FK를 성능이 우수한 순서대로 칼럼의 순서를 조정해야 함

6. 성능 관점에서의 데이터 모델 검증
