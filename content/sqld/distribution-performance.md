분산 데이터베이스란, 데이터베이스를 **물리적으로 분산**(여러 지역, 여러 노드로 위치) 시킨 후 빠른 **네트워크 환경**을 이용해 논리적으로 **하나의 가상 시스템**으로 사용하는 데이터베이스이다.

# 분산 데이터베이스의 투명성(Transparency)

> 분산 데이터베이스가 되기 위해서는 6가지 투명성(Transparency)을 만족해야 하지만 실제로 모두 만족하는 사례는 잘 없음

1. **위치 투명성**
   - 실제 위치에 상관없이 논리적인 명칭만으로 액세스 가능
2. **중복 투명성**
   - DB 객체가 여러 위치에 중복 되어 있는지 알 필요가 없는 성질
3. **장애 투명성**
   - 구성요소(DBMS, Computer)의 장애에도 트랜잭션의 원자성 유지
4. **병행 투명성**
   - 다수 트랜잭션 동시 수행 시 결과의 일관성 유지, 타임스탬프, 분산 2단계 Locking을 이용 구현
5. 지역사상 투명성
   - 지역 DBMS와 물리적 DB사이의 매핑 보장. 각 지역 시스템 이름과 무관한 이름 사용 가능
6. 분할 투명성 (단편화)
   - 하나의 논리적 Relation이 여러 단편으로 분할되어 각 단편의 사본이 여러 위치에 저장

# 분산 데이터베이스의 적용 방법 및 장단점

분산 데이터베이스는 업무 구성에 따른 아키텍처 특징에 따라 구성해야 한다. 즉, 업무 특성에 맞게 구성해야 한다.

## 분산 데이터베이스 장단점

> 데이터베이스 분산구성의 가치는 **원거리 네트워크 부하 절감**과 **트랜잭션이 집중되는 데이터베이스를 분산** 처리하여 빠른 성능을 제공하는 것에 있음

- **장점**
  - 지역 자치성이 높음
  - 자료의 공유성이 향상됨
  - 분산 제어 가능
  - 시스템 성능 향상
  - 중앙 컴퓨터의 장아개 전체 시스템에 영향 X
  - 효용성, 융통성, 신뢰성, 가용성
- **단점**
  - 용량 확장이 용이함
  - DBMS가 수행할 기능이 복잡
  - 설계가 어려움
  - 개발, 처리 비용 증가
  - 잠재적 오류 증가

## 분산 데이터베이스의 적용 기법

분산 데이터베이스는 일단 통합 데이터 모델링을 하고, 각 테이블별로 업무적인 특징에 따라 설계한다.

### 테이블 위치 분산

> 정보를 이용하는 형태가 각 위치별로 차이가 있을 경우에 이용

- 설계 된 테이블의 위치를 각각 다르게 함
- 테이블 구조 변화 **없음**
- 다른 데이터베이스에 중복 생성되지 않음
- 테이블의 위치를 도식화한 데이터베이스 문서 필요

예를 들어 자재품목은 본사에서 구입하여 관리하고 각 지사별로 자재품목을 이용하여 제품을 생산한다고 하면, 본사 DB에는 자재품목 테이블을 두고 자사 DB에는 생산제품 테이블을 두는 것이다.

### 테이블 분할(Fragmentation) 분산

> Row 단위 분할의 **수평 분할**과 Column 단위 분할의 **수직 분할**이 있음

- **수평 분할 (Horizontal Fragmentation)**
  - 특정 컬럼의 **값**을 기준으로 로우(Row)를 분리
    - 지사(Node)별로 사용 Row가 다를 때 유용
  - 모든 데이터가 각 지사별로 분리되어 있는 형태
    - 자신의 데이터에 대해 수정
  - 데이터를 한군데 집합시켜 놓아도 의해 중복이 발생하지 않음
    - 단, 통합 처리 시 Join으로 인한 성능저하가 예상됨
    - 따라서 통합 처리가 많지 않은 경우 사용
    - 데이터 무결성은 보장됨
- **수직 분할 (Vertical Fragmentation)**
  - 테이블 컬럼을 분리
  - 데이터가 각 지사별로 분리되어 있는 형태
    - 지사별로 쪼개어진 테이블들을 조합하면 하나의 완전한 테이블이 구성되어야 함
    - 데이터 중복은 발생되지 않음
  - 동일한 Primary Key구조와 값을 가지고 있어야 함
    - 단, 통합 처리 시 Join으로 인한 성능저하가 예상됨
    - 따라서 통합 처리가 많지 않은 경우 사용
  - 예시
    - 제품의 재고량은 각 지사별로 관리
    - 제품에 대한 단가는 본사에서 관리

참고) 수평 분할과 수직 분할의 내용은 [여기](/database/sqld/large-data)에서도 다루었음

### 테이블 복제(Replication) 분산

> 동일한 테이블을 다른 지역이나 서버에서 동시에 생성하여 관리하는 유형임

- **부분 복제 (Segment Replication)**
  - 실제 프로젝트에서 많이 사용하는 데이터베이스 분산 기법
  - 통합된 테이블을 한군데(본사)에 둠
  - 지사(Node)별로 각 지사의 Row를 중복으로 가짐
    - 지사 간에는 데이터의 중복이 발생하지 않음
    - 본사와 지사 간에는 데이터 중복이 항상 발생
  - 통합 처리 시 통합 테이블을 이용하므로 Join이 발생하지 않음 (빠른 처리)
  - 보통 지사 데이터의 수정이 발생하여 본사로 복제해 데이터 정합성 유지
- **광역 복제 (Broadcast Replication)**
  - 통합 테이블을 모두가 복제하여 동일한 데이터를 가짐
  - 본사나 지사나 데이터 처리에 특별한 제약을 받지 않음
  - 보통 본사에서 데이터가 입력/수정/삭제 되고 지사에서는 이를 이용함
  - 예시
    - 본사에서 코드 테이블에 데이터에 대해 입력, 수정, 삭제
    - 각 지사에서는 복제한 코드 데이터를 이용하여 프로세스

두 기법 모두 데이터를 복제(Replication)하는데 많은 시간이 소요되고 데이터베이스와 서버에 부하(Load)가 발생하므로 보통 실시간(On-Line) 처리에 의해 복사하는 것보다는 **배치에 의해 복제**가 되도록 한다.

### 테이블 요약(Summarization) 분산

동일한 테이블 구조를 가지고 있으면서 분산되어 있는 동일한 내용의 데이터를 이용하여 통합된 데이터를 산출하는 방식의 **분석 요약 (Rollup Summarization)** 과 분산되어 있는 다른 내용의 데이터를 이용하여 통합된 데이터를 산출하는 방식의 **통합 요약 (Consolidation Summarization)** 이 있다.

**분석 요약 예시**  
A, B, C 각 지사는 모두 'P001' 상품의 판매 실적을 가지고 있고 각 판매 실적은 서로 다르므로 요약 정보들을 본사가 통합하여 'P001' 상품의 전체 판매실적을 산출한다.

**통합 요약 예시**  
A는 'P001', B는 'P002', C는 'P003' 상품만을 취급한다. 각 상품에 대한 요약 정보들을 본사가 통합하여 통합 판매실적을 산출한다.

두 기법 모두 일반 업무가 종료되는 야간에 수행하여 생성하는 것이 일반적이다.

# 분산 데이터베이스를 적용하면 좋은 경우

- 성능이 중요한 사이트
- 공통코드, 기준정보, 마스터 데이터
- 실시간 동기화가 요구되지 않는 경우
  - Near Real Time의 업무적인 특징을 가지고 있을 때도 구성 가능
- 특정 서버에 부하가 집중이 되어 부하를 분산해야하는 경우
- 백업 사이트(Disaster Recovery Site)를 구성하는 경우
