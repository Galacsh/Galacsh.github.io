대량의 데이터가 하나의 테이블에 집약되어 있고 하나의 하드웨어 공간에 저장되어 있으면 성능저하를 피하기 힘들다. 중요한 업무 데이터가 특정 테이블에 집중되는 경우, 테이블 단위에서 분할의 방법을 적용할 필요가 있다.

# 대량 데이터 성능 저하 원인

**1. 한 테이블에 데이터가 대량으로 집중**

- 인덱스의 트리 구조가 너무 커져 효율성 저하
  - 인덱스 용량 증가
  - 인덱스를 찾아가는 단계가 깊어져 조회 성능 저하

**2. 한 테이블에 너무 많은 수의 컬럼 존재**

- 데이터가 디스크의 여러 블록에 존재하여 I/O 증가
  - 컬럼이 많아지면 물리 디스크 여러 블록에 저장
  - SQL 문장이 여러 블록에 I/O 하며 성능 저하
  - **로우 체이닝(Row Chaining)** 증가
    - 로우 길이가 너무 길어서 데이터 블록 하나에 데이터가 모두 저장되지 못하고 두 개 이상의 블록에 걸쳐 저장되는 현상
  - **로우 마이그레이션(Row Migration)** 증가
    - 데이터 블록에서 수정이 발생하면 수정된 데이터를 해당 데이터 블록에 빈 공간이 없어서 저장되지 못하고 새로운 블록을 할당 받아 옮겨진 뒤 수정되는 현상

# 테이블 분할

> 테이블 분할 방법에는 크게 수평 분할과 수직 분할로 나눌 수 있음

## 테이블 분할 절차

1. 데이터 모델링 완성
2. 데이터베이스 용량 산정
   - 어느 테이블에 데이터의 양이 대용량이 되는지 분석하는 것
3. 트랜잭션 처리 패턴 분석
4. 집중 처리가 컬럼 단위로 발생하는지 로우 단위로 발생하는지 분석
   - 컬럼 단위 집중되는 경우
     - 테이블을 1:1 형태로 분리할 수 있는지 검증
   - 로우 단위 집중되는 경우
     - 테이블 파티셔닝 전략 고려

## 수직 분할

> 트랜잭션이 발생할 때 어떤 컬럼에 집중되는지 분석하여 테이블을 분리하면 디스크 I/O가 감소하게 되어 성능이 개선됨

테이블에 컬럼 수가 아주 많은 경우 여러 블록에 데이터가 산재되므로 디스크 I/O가 많이 발생한다.

```
OOOO | OO
OOOO | OO
OOOO | OO
OOOO | OO
```

## 수평 분할

> 논리적으로는 하나의 테이블로 보이지만 물리적으로 여러 개의 테이블로 쪼개어 저장될 수 있는 파티셔닝을 적용함

데이터량이 몇 천만건을 넘어서면 아무리 서버 사양이 훌륭하고 인덱스를 잘 생성해준다고 하더라도 SQL 문장의 성능이 나오지 않는다.

```
OOOOOO
OOOOOO
------
OOOOOO
OOOOOO
```

### RANGE PARTITION

> 대상 테이블이 날짜 또는 숫자 값으로 분리가 가능하고 각 영역별로 분리함

데이터보관주기에 따라 테이블에 데이터를 쉽게 지우는 것이 가능하므로(파티션 테이블을 DROP하면 되므로) 데이터보관주기에 다른 테이블관리가 용이하다.

### LIST PARTITION

> 핵심적인 코드값 등으로 PK가 구성되어 있고 대량의 데이터가 있는 테이블이라면 값 각각에 대해 분리함

대용량 데이터를 특정값에 따라 분리 저장할 수는 있으나 RANGE PARTITION과 같이 데이터 보관주기에 따라 쉽게 삭제하는 기능은 제공될 수 없다.

### HASH PARTITION

> 지정된 HASH 조건에 따라 해슁 알고리즘이 적용되어 테이블을 분리함

설계자는 테이블에 데이터가 정확하게 어떻게 들어갔는지 알 수 없으며 역시 성능향상을 위해 사용하며 데이터 보관주기에 따라 쉽게 삭제하는 기능은 제공될 수 없다.
