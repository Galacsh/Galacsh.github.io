# 데이터베이스

일반적으로 데이터베이스는 특정 기업이나 조직 또는 개인이 필요에 의해 데이터를 일정한 형태로 저장해 놓은 것을 의미한다.

## 데이터베이스의 발전

- 1960년대
  - 파일 구조로 데이터 저장 및 관리
- 1970년대
  - 계층형(Hierarchical) 데이터베이스, 망형(Network) 데이터베이스 등 상용화
- 1980년대
  - 관계형 데이터베이스 상용화
- 1990년대
  - Oracle, Sybase, DB2, SQL Server 외 많은 제품들이 보다 향상된 기능으로 핵심 솔루션화
  - 객체 지향 정보를 지원하기 위해 객체 관계형 데이터베이스로 발전

## 관계형 데이터베이스 (Relational Database)

> 1970년 대 E.F. Codd 박사 논문에서 첫 소개되었고,
> 이후 파일 시스템, 계층형/망형 데이터베이스를 대체하였다.

- 정규화
  - 이상(Anomaly) 현상 제거
  - 데이터 중복 최소화
- 동시성 관리
- 병행 제어
  - 많은 사용자들이 동시에 데이터 공유/조작
- 메타 데이터 관리
  - 데이터의 성격, 속성 또는 표현 방법 등 체계화
- 데이터 표준화를 통한 데이터 품질 확보
- 보안 기능 제공 (인증된 사용자만 가능)
- 다양한 제약 조건을 통해 사용자의 실수 방지 및 데이터 무결성 보장
- 시스템 장애에 대비한 회복/복구 기능

# SQL (Structured Query Language)

> 관계형 데이터베이스에서 데이터 정의, 데이터 조작, 데이터 제어를 하기 위해 사용하는 언어

- 관계형 데이터베이스와 사용자가 대화를 할 수 있게 함
  - 사용자는 SQL을 통해 특정 데이터들의 집합에서 조회/입력/수정/삭제
- ANSI/ISO 표준이 있음
- RDBMS 벤더마다 차별화 기능이 있음
  - 편리성을 위한 추가 기능, 내장 함수 등
  - 이런 기능(함수)를 사용할 지, 표준을 따를지는 사용자 몫임
    - 단, 표준을 기준으로 할 것을 권고함

이러한 SQL의 튜닝 목적은 시스템에 큰 영향을 주는 SQL을 가장 효과적(응답시간, 자원 활용 최소화)으로 작성하는 것이며, 참고로 SQL의 최초 이름이 SEQUEL(Structured English QUEry Language)이었기 때문에 ‘시큐얼’로 읽는 경우도 있지만, 표준은 SQL이므로 ‘에스큐엘’로 읽는 것을 권고한다.

- SQL 명령어의 종류
  - **데이터 조작어 (DML, Data Manipulation Language)**
    - SELECT / INSERT / UPDATE / DELETE
    - 데이터베이스 테이블 내 데이터를 조작하는 명령어들을 말함
  - **데이터 정의어 (DDL, Data Definition Language)**
    - CARD → CREATE / ALTER / RENAME / DROP
    - 데이터 구조를 정의하는 명령어들 (데이터 구조 생성, 변경, 이름 변경, 삭제)
  - **데이터 제어어 (DCL, Data Control Language)**
    - RG → REVOKE / GRANT
    - 데이터베이스에 접근하고 객체들을 사용하도록 권한을 주고 회수하는 명령어들
  - **트랜잭션 제어어 (TCL, Transaction Control Language)**
    - 트랜잭션 하면, COMMIT / ROLLBACK / SAVEPOINT
    - 논리적인 작업의 단위를 묶어서 제어하는 명령어들

SQL 명령어는 3가지 그룹인 DDL, DML, DCL로 나눌 수 있는데, TCL의 경우 굳이 나눈다면 일부에서 DCL로 분류하기도 하지만, 다소 성격이 다르므로 별도의 4번째 그룹으로 분리할 것을 권고한다.

# 테이블

> 테이블은 데이터를 저장하는 객체(Object)로서 관계형 데이터베이스의 기본 단위

- 용어
  - 테이블
    - 행과 컬럼의 2차원 구조를 가진 데이터의 저장 장소이며, 데이터베이스의 가장 기본적인 개념
  - 컬럼/열
    - 2차원 구조를 가진 테이블에서 세로 방향으로 이루어진 하나하나의 특정 속성 (더이상 나눌 수 없는 특성)
  - 행
    - 2차원 구조를 가진 테이블에서 가로 방향으로 이루어진 연결된 데이터
  - 정규화
    - 테이블을 분할하여 데이터의 정합성을 확보하고, 불필요한 중복을 줄이는 프로세스
  - 기본키
    - 테이블에 존재하는 각 행을 한 가지 의미로 특정할 수 있는 한 개 이상의 컬럼
  - 외부키
    - 다른 테이블의 기본키로 사용되고 있는 관계를 연결하는 컬럼

# ERD (Entity Relationship Diagram)

ERD(Entity Relationship Diagram)는 관계의 의미를 직관적으로 표현할 수 있게 테이블 간 서로의 상관 관계를 그림으로 도식화한 것이다.

ERD의 구성 요소는 **엔터티 (Entity)**, **관계 (Relationship)**, **속성 (Attribute)** 3가지이며 현실 세계의 데이터는 이 3가지 구성 요소로 모두 표현이 가능하다.

표기법으로는 IE(Information Engineering) 표기법과 Barker 표기법이 있다.

- ERD 표기법
  - 실선 : 식별 관계
    - 부모 테이블의 기본키를 자식 테이블이 가지고 있으며 이를 기본키로 사용함
  - 점선 : 비식별 관계
    - 부모 테이블의 기본키를 자식 테이블이 가지고 있지만 기본키로 사용하지 않음
  - 까마귀 발 : 대응되는 대상이 여럿 → n > 1
  - 원 : Optional → n == 0
  - 수직바 : Mandatory → n == 1

까마귀발, 원, 수직바 등의 조합은 '또는'으로 생각하면 된다.  
원 + 수직바 → n == 0 or n == 1

ERD, IE와 Barker 표기법이 잘 정리된 글은 참고 자료에서 확인할 수 있다.
