# 정규화

> 관계형 데이터베이스의 설계에서 중복을 최소화하게 데이터를 구조화하는 프로세스

정규화의 목적은 다음과 같이 정리할 수 있다.

- **중복 최소화**
- 데이터 이상(Data Anomaly) 발생 방지
  - **삽입 이상**
    - 데이터 삽입 시 불필요한 정보를 함께 저장하지 않고서는 저장하는 것이 불가능
  - **삭제 이상**
    - 데이터 삭제 시 꼭 필요한 값도 삭제
  - **갱신 이상**
    - 데이터 갱신 시 중복 데이터 중 일부만 갱신되어 불일치 발생 
- 데이터베이스 구조 변경 시 **재디자인 최소화**
- **다양한 관점의 질의** 지원

## 배경 지식

> 아래 내용에서 대문자는 속성(컬럼)의 집합을, 소문자는 속성(컬럼)을 의미함

- **함수적 종속성 (FD, Functional Dependency)**
  - 어느 속성군의 값이 정해지면 다른 속성군의 값이 정해지는 것
  - ex. 직원의 생일은 누구냐(직원 ID)에 의해 정해짐
    - {직원 ID} → {직원 생일} 
    - 직원 ID가 결정자, 직원 생일이 종속자
- **완전 함수 종속**
  - {a, b, c} → {d}일 때, {b, c} → {d} 처럼 부분 집합에 대한 FD가 없는 것
- **이행 함수 종속**
  - = 어떤 속성군이 건너 건너 다른 속성군의 값을 정할 수 있음
  - A → B, B → C일 때, A → C이면 이행적 함수 종속
- **다치(다중값) 종속 (MVD, Multiple Value Dependency)**
  - 함수적 종속성이 다치 종속의 일종임
  - 어떤 레코드의 존재가 다른 레코드의 존재로 이어짐을 의미
  - {a, b, c}가 있다고 할 때, a-b, a-c는 연관성이 있으나 b-c는 연관성이 없는데도 PK인 경우를 생각하면 됨 
  - 머리가 두개인 화살표로 표기함
    - A ->> B = B가 A에 다치 종속됨

**다치 종속 예시**

| 학생   | 수강과목 | 동호회 |
| ----- | ------- | ----- |
| 홍길동 | 수학    | 스키반 |
| 홍길동 | 수학    | 서도회 |
| 홍길동 | 물리학  | 스키반 |
| 홍길동 | 물리학  | 서도회 |
| 홍길동 | 영어    | 스키반 |
| 홍길동 | 영어    | 서도회 |

학생 → 수강과목, 학생 → 동호회이지만 수강과목과 동호회는 아무 연관이 없는 상황임에도
 "홍길동"이 새로운 동호회에 가입하기 위해서는 3개의 행이 추가되어야 함

- **조인 종속 (JD, Join Dependency)**
  - 테이블 R이 R의 속성의 부분집합을 가지는 여러 개의 테이블들을 조인하여 만들어질 수 있는 경우 (무손실분해)
- **슈퍼키**
  - 레코드를 유일하게 식별해낼 수 있는 속성들의 집합
  - 즉, 한 테이블은 여러 슈퍼키를 가질 수 있음
  - 유일성 O, 최소성 X
- **후보키**
  - 식별에 필요없는 속성을 제거한 슈퍼키의 부분 집합
  - 유일성 O, 최소성 O
- **기본키**
  - 후보키들 중 하나
  - null 값을 가질 수 없음
  - 단순한 후보 키중 값이 자주 변경되는 속성이 없는 것 선택
- **대체키**
  - 기본키를 제외한 후보키의 부분 집합
- **무결성 제약조건**
  - **개체 무결성**
    - 기본키는 중복 불가, 각 속성은 NULL 값 불가
  - **참조 무결성**
    - 외래키 값은 NULL 또는 참조 릴레이션의 기본키 값과 동일해야 함
  - **도메인 무결성**
    - 속성 값은 그 속성이 정의된 도메인 내 값이어야 함
  - **고유 무결성**
    - 특정 속성에 고유한 값을 가지도록 조건이 주어지면, 그 속성값은 모두 달라야 함
  - **NULL 무결성**
    - 특정 속성값에 NOT NULL 조건이 주어지면, NULL이 올 수 없음
  - **키 무결성**
    - 한 릴레이션(테이블)에는 최소 1개의 키가 존재해야 함
- **드라이빙 테이블**
  - Join 시 먼저 액세스 돼서 Access Path를 주도하는 테이블
  - 나중에 액세스 되는 테이블을 Driven 테이블이라고 함

## 정규형 소개

> 비공식적으로 제 3 정규화가 되었으면 정규화 되었다고 판단한다.

3NF 테이블 대부분이 데이터 이상(Data Anomaly)이 없고 이미 BCNF, 4NF, 5NF를 만족하는 경우가 많기 때문이다.

### 제 1 정규형 (1NF)

> 모든 도메인이 업무적인 의미로 원자값(Atomic Value)으로 구성되어 있음

**제 1 정규형이 아닌 예시 (1)**  
다음 경우는 홍길동의 전화번호가 2개이므로 구분자 '^'를 이용해 **하나의 컬럼에 두 값**을 저장하는 모습이다.

| 유저아이디 | 이름  | 전화번호                      |
| -------- | ----- | --------------------------- |
| hello123 | 홍길동 | 010-1111-1111^010-2222-2222 |
| asdf     | 가나다 | 321-7654-0198               |

**제 1 정규형이 아닌 예시 (2)**  
다음 경우는 동일한 도메인과 의미를 갖는 중복되는 컬럼을 가지는 경우이다. 

| 유저아이디 | 이름  | 전화번호 1      | 전화번호 2     |
| -------- | ----- | ------------- | ------------- |
| hello123 | 홍길동 | 010-1111-1111 | 010-2222-2222 |
| asdf     | 가나다 | 321-7654-0198 |               |

따라서 다음과 같이 2개로 분리해야 한다.

| 유저아이디 | 이름  |
| -------- | ----- |
| hello123 | 홍길동 |
| asdf     | 가나다 |

| 유저아이디 | 전화번호       |
| -------- | ------------- |
| hello123 | 010-1111-1111 |
| hello123 | 010-2222-2222 |
| asdf     | 321-7654-0198 |

### 제 2 정규형 (2NF)

> 제 1 정규형을 만족하고 부분 함수 종속이 없는 상태

예를 들어, 다음과 같은 테이블이 있을 때,

- 종업원의 기술 테이블
  - **종업원** | **기술** | 근무지

근무지는 종업원에 의해서 결정되는 것이지 종업원이 어떤 기술을 가지냐에 따라 근무지가 정해지는 것이 아니다. 따라서 다음과 같이 분리하여 부분 함수 종속성을 제거할 수 있다.

- 종업원 테이블
  - **종업원** | 근무지
- 종업원의 기술 테이블
  - **종업원** | **기술**

### 제 3 정규형 (3NF)

> 제 2 정규형을 만족하고 기본키 외의 속성들 간에 함수적 종속성을 가지지 않도록 이행 함수 종속성을 제거한 상태

예를 들어, 다음과 같은 테이블이 있을 때,

- 대회 우승자 테이블
  - **대회** | **연도** | 우승자 | 우승자 생년월일

해당 연도의 대회는 우승자를 결정할 수 있고 우승자가 결정되니 자연스레 우승자 생년월일도 결정하는 상황이다. 우승자와 우승자 생년월일은 기본키 외의 속성들인데 함수적 종속성이 나타나고 있다. 따라서 다음과 같이 테이블을 분리하여 이행적 함수 종속성을 제거할 수 있다.

- 대회 우승자 테이블
  - **대회** | **연도** | 우승자
- 우승자 생년월일 테이블
  - **우승자** | 우승자 생년월일

### 보이스-코드 정규형 (BCNF)

> 제 3 정규형인 상태에서 키에 속하지 않는 속성군에서 결정자 속성군으로의 함수적 종속성 제거하여, 모든 결정자가 후보 키인 상태

예를 들어, 다음과 같은 테이블이 있을 때,

- 수강 과목
  - **학번** | **과목** | 교수
    - 교수 → 과목 (교수가 한 과목만 담당)

교수가 정해지면 과목이 정해지더라도 제 3 정규형인 것은 변함 없다. 기본키가 아닌 속성 간의 함수적 종속성이 있는 것은 아니기 때문이다. 

하지만 새로운 교수가 추가되어 담당 과목이 정해졌을 때, 수강할 학생이 없다면 삽입에 문제가 생긴다. 따라서 모든 결정자가 후보 키일 수 있도록 테이블을 분리해주면 다음과 같다.

- 수강 과목 테이블
  - **학번** | **과목**
- 교수의 담당 과목 테이블
  - **교수** | 과목

### 제 4 정규형 (4NF)

> 2NF, 3NF, BCNF까지는 함수적 종속성과 관련된 내용이었다면 제 4 정규형은 조금 더 일반적인 종속성인 다치 종속성을 제거한 상태임

예를 들어, 다음과 같은 테이블이 있을 때,

- 피자 배달 가능 지역 테이블
  - **음식점** | **피자 종류** | **배달 지역**

특정 음식점의 어떤 피자가 어떤 지역에 배달될 수 있는지 알 수 있을 것이다. 실제로 배달 지역에 따라 배달할 수 있는 피자 종류가 다르다면 딱히 문제가 있는 테이블로 볼 수 없다.

하지만 배달 지역은 결국 음식점에 의해 결정되고 그 음식점의 어떤 피자인지는 상관이 없다면 위 테이블은 중복되는 정보(다중값)를 많이 포함하게 된다.

따라서 다음과 같이 수정하여 다치 종속성을 제거할 수 있다.

- 음식점 별 피자 종류 테이블
  - **음식점** | **피자 종류**
- 음식점 별 배달 가능 지역 테이블
  - **음식점** | **배달 지역**

### 제 5 정규형 (5NF)

> 모든 조인 종속(JD)이 후보키를 통해서만 만족되는 상태이거나 더 이상 무손실 분해 될 수 없는 상태

A, B, C가 있을 때 A-B, A-C, B-C는 연관성이 있으나, A-B-C는 연관성이 없는데도 불구하고 하나의 테이블에 PK로 사용이 될 때를 제 5 정규화가 필요한 경우라고 할 수 있다.

# 반정규화

> 정규화된 엔터티, 속성, 관계에 대해 시스템의 성능향상과 개발(Development)과 운영(Maintenance)의 단순화를 위해 중복, 통합, 분리 등을 수행하는 데이터 모델링의 기법을 의미함

데이터 무결성이 깨질 수 있는 위험을 무릅쓰고 데이터를 중복하여 반정규화를 적용하는 이유는 데이터를 조회할 때 디스크 I/O량이 많아서 성능이 저하되거나 경로가 너무 멀어 조인으로 인한 성능저하가 예상되거나 칼럼을 계산하여 읽을 때 성능이 저하될 것이 예상되는 경우 반정규화를 수행한다.

## 반정규화 적용 방법

> 업무적으로 조회에 대한 처리성능이 중요한 경우, 함수적 종속 관계는 위반하지 않지만 데이터의 중복성을 증가시켜야만 데이터 조회의 성능을 향상시키는 경우 부분적으로 반정규화를 고려함

데이터 무결성을 보장할 수 있는 방법을 고려한 이후에 반정규화를 적용

**1. 반정규화 대상 조사**

- 자주 사용되는 테이블에 접근하는 프로세스 수가 많고 항상 일정한 범위만을 조회할 때
- 대량의 데이터 범위를 자주 처리하는 경우에 처리 범위를 일정하게 줄여야 할 때
- 통계 정보를 필요로 할 때 별도의 통계테이블(반정규화 테이블)을 생성
- 많은 조인(JOIN)이 걸려 데이터를 조회하는 작업이 기술적으로 어려울 경우

**2. 다른 방법 검토**

- 뷰(View) 테이블 사용
  - 조회 성능을 향상시키지는 못함
  - 개별자마다 다른 SQL 문장으로 성능이 저하되는 것은 막을 수 있음
- 클러스터링 적용
  - 대량의 데이터를 특정 클러스터링 팩트에 의해 저장방식을 다르게 함
  - 입력/수정/삭제 시에는 성능이 저하되므로 조회 중심일 때 사용
  - 인덱스를 통해 성능 향상이 불가할 때 사용
- 인덱스 조정
- 파티셔닝 기법
- 응용 애플리케이션 로직 변경 (ex, 캐싱)

**3. 반정규화를 적용한다.**

- 테이블 반정규화
- 속성 반정규화
- 관계 반정규화

## 반정규화 기법

**1. 테이블 반정규화**

| 기법 분류   | 기법                 | 내용                       |
| --------- | -------------------- | ------------------------- |
| 테이블 병합 | 1:1 관계 테이블병합    | 1:1 관계를 통합하여 성능향상 |
|           | 1:M 관계 테이블병합     | 1:M 관계 통합하여 성능향상 |
|           | 슈퍼/서브타입 테이블병합 | 슈퍼/서브 관계를 통합하여 성능향상 |
| 테이블 분할 | 수직 분할             | 컬럼 단위의 테이블을 디스크 I/O를 분산처리 하기 위해 트랜잭션 유형을 파악하여 테이블을 1:1로 분리하여 성능향상 |
|           | 수평 분할             | 로우 단위로 집중 발생되는 트랜잭션을 분석하여 로우 단위로 테이블을 쪼갬 |
| 테이블 추가 | 중복 테이블 추가       | 다른 업무이거나 서버가 다른 경우 동일한 테이블 구조를 중복하여 원격 조인을 제거 |
|           | 통계 테이블 추가       | SUM, AVG 등을 미리 계산해두어 조회 시 성능 향상 |
|           | 이력 테이블 추가       | 마스터 테이블에 존재하는 레코드를 중복하여 이력 테이블에 존재하는 방법도 반정규화 유형 중 하나임 |
|           | 부분 테이블 추가       | 전체 컬럼 중 집중 이용되는 컬럼들이 있을 때 디스크 I/O를 줄이기 위해 별도로 컬럼을 모아놓은 테이블 생성 |

‎

**2. 속성(컬럼) 반정규화**

| 반정규화 기법                    | 내용 |
| ------------------------------ | --- |
| 중복 컬럼 추가                   | 조인을 감소시키기 위해 중복 컬럼을 추가함 |
| 파생 컬럼 추가                   | 트랜잭션 처리 중 계산 과정에 의한 성능저하를 줄이기 위해 계산된 값을 컬럼에 보관함 |
| 이력 테이블 컬럼 추가             | 이력 테이블의 성능저하를 줄이기 위해 (ex. 특정 일자 조회) 기능성 컬럼 추가 (시작일자, 종료일자 등) |
| PK에 의한 컬럼 추가              | 복합 의미를 갖는 PK를 단일 속성으로 구성하여 조회 시 성능저하가 발생하는 경우 해당 속성을 일반 속성으로 추가함 |
| 응용 시스템 오작동을 위한 컬럼 추가 | 업무적으로는 의미가 없지만 데이터 복구용으로 임시적으로 중복 보관하는 컬럼 |

‎

**3. 관계 반정규화**

데이터를 처리하기 위해 여러 경로를 거쳐 조인 할 때 발생할 수 있는 성능 저하를 예방하기 위해 추가적인 관계를 맺는 방법으로 데이터 무결성을 꺠뜨릴 위험을 갖지 않고서도 데이터 처리 성능을 향상시킬 수 있는 반정규화 기법을 **중복 관계 추가**라고 한다.

# 성능 관점에서의 정규화 / 반정규화

> 정규화를 잘하되, 전략적으로 반정규화를 진행하여 조인으로 인한 성능 저하를 방지하자.

- 보통 정규화를 잘하게 되면 성능이 향상되는 특징이 있음
  - 일반적으로 정규화는 데이터의 중복을 제거함
    - 드라이빙 되는 대상 테이블의 데이터를 줄일 수 있음
  - 또한 데이터는 보통 관심사별로 처리되는 경우가 많기 때문
- 단, 정규화를 할수록 엔터티가 계속 생겨 조인이 많아지므로 성능저하가 발생하기도 함
  - 정규화를 통해 성능이 향상되는 경우도 있으므로 고정관념 X

예를 들어, 동일한 속성을 한 테이블에 속성1, 속성2, ... 과 같이 모델링하는 것도, 조회 성능을 올리려면 각 속성에 인덱스를 만들어야 하는데, 정규화를 하면 데이터 양을 줄일뿐더러 인덱스를 추가로 생성하지 않아도된다.

참고로 인덱스가 많아지면 그만큼 입력/수정/삭제 처리가 느려지기도 하며, 일반 업무 처리 시에는 인덱스 수를 7~8개가 넘지 않도록 하는 게 좋다.

## 정규화 테이블 성능 저하 케이스

다음과 같은 데이터 모델에서 최신 공급자 정보를 가져온다고 생각해보자.

- **공급자**
  - 공급자번호 (PK) | 공급자명
- **전화번호**
  - 공급자번호 (PK, FK) | 순번 (PK) | 전화번호
- **메일주소**
  - 공급자번호 (PK, FK) | 순번 (PK) | 메일주소

공급자의 최신 전화번호, 메일 주소를 함께 뽑아내려면 복잡한 SQL 구문이 생길 것이다.
따라서 공급자 테이블을 적절히 반정규화를 하게 되면

- **공급자**
  - 공급자번호 (PK) | 공급자명 | 전화번호 - 최근 값 | 메일주소 - 최근 값

조인이 발생하지 않아 성능 향상되는 점과 간단한 SQL 구문을 얻을 수 있다.

이 외에도, 여러 데이터베이스 서버에 저장된 분산 데이터베이스 환경에서도 필요한 중복 컬럼을 생성하여 서버간 조인을 하지 않도록 하여 성능을 향상시키기도 한다.
